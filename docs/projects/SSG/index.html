Introduction and motivation

The blog you are currently reading is built and generated by an SSG I built as a project. In this page I am going to show you how I built it step by step.

What is an SSG?

A Static Site Generator (SSG) is a tool that takes raw content files (markdown files and images) and turns them into a static website. It makes it possible for its' users to create their own static websites withouth ever writing a line of html code! Here is a link to the Github repository, where is the source code for this project.

What are the elements of an SSG?

The SSG tool is loosely divided into two main parts:

  • a converter of raw markdown text to html code
  • a generator of webpages, which combines the converted html code, raw images and an example css file for formating into a website fully ready to be hosted online

Here is shown the logical steps my code needed to accomplish in a high level view

A conversion algorithm of markdown to html, needs to accomplish the following steps:

  1. divide the markdown code into blocks, depending on their type (heading, unordered list, ordered list, code, quote, paragraph) and classify them
  2. for each block identify the inline elements (bold, italic and code text as well as images and links)
  3. convert the inline elements into html code and pass the html code to the blocks
  4. convert the overall block into html syntax

Generator of webpages:

  1. combine a markdown file (representing one page) with the images used and a css file for formating into one html page
  2. do the same recursevely for all markdown files to generate the whole website

Now I'll dive deeper in how specifically I built the two main parts of the SSG

How I built the converter of markdown to html

I used tree-like data structure using nodes in the conversion steps. The flow of information follows:

  • Raw markdown -> TextNode -> HTMLNode -> html syntax

TextNode is an objects which contains information about the inline elements, once they are identified and stores it in terms of raw text, text type and optionally url variable (where links and image source is stored). Here is how the constructor of a TextNode looks like in the code:

class TextNode():
def __init__(self, text, text_type, url = None):
self.text = text
self.text_type = text_type
self.url = url

The TextNode objects have a method that converts them to HTMLNode objects. The HTMLNode objects each represent a node in a tree data structure, where each node holds information about an html segment - tag, value and optionally props field, where attributes to the tag are saved, if there are any, as well as reference to the children nodes (the inner html segments of the chosen), if there are any:

class HTMLNode():
def __init__(self, tag=None, value=None, children=None, props=None):
self.tag = tag
self.value = value
self.children = children
self.props = props

I use two chidclasses of the HTMLNode class, namely ParentNode and LeafNode. LeafNode represents, unsurprisingly, the leaf nodes of the data structure (those without children) and the ParentNode represents all other nodes - those with children (inner html segments). The LeafNode has a method that converts them into html syntax:

The ParentNode also has its own method for conversion to html syntax. It utilizes the fact that it must not have value (since the would be value is transfered to its children) and must have a tag (otherwise it would be just raw text, which is only possible for a LeafNode) and children:

def generate_pages_recursively(dir_path_content, template_path, dest_dir_path, basepath):
